# 认证与权限设计

## 1. 目标

本文档定义第一阶段的认证与权限方案，目标是让系统从“匿名项目编辑器”升级到“团队协作平台”。

第一阶段重点解决：

- 用户身份识别
- 工作区成员权限
- 项目成员权限
- API 侧权限校验

第一阶段暂不解决：

- SSO
- OAuth 第三方登录
- API Token
- 复杂自定义策略引擎

## 2. 认证策略

## 2.1 推荐方案

第一阶段建议使用：

- 邮箱 + 密码登录
- 服务端签发 session 或 JWT
- 前后端通过受保护接口识别当前用户

如果前后端是同域部署，推荐优先考虑：

- `HttpOnly Cookie + Session`

原因：

- 对浏览器应用更直接
- 安全边界更清晰
- 后续扩展刷新机制更自然

如果后续存在跨域或多客户端需求，再评估 JWT。

## 2.2 第一阶段必须具备的认证能力

- 用户注册
- 用户登录
- 用户登出
- 获取当前用户信息
- 校验未登录状态

## 2.3 用户信息

第一阶段用户信息最少包含：

- 邮箱
- 显示名
- 密码哈希
- 是否激活

后续可扩展：

- 头像
- 组织资料
- 第三方登录绑定

## 3. 权限分层

权限分两层：

- 工作区级权限
- 项目级权限

这是必要的，因为：

- 工作区管理的是“团队协作边界”
- 项目管理的是“具体文档协作边界”

## 4. 工作区权限模型

### 4.1 工作区角色

- `owner`
- `admin`
- `member`
- `viewer`

### 4.2 工作区角色说明

#### owner

权限：

- 修改工作区设置
- 邀请成员
- 修改成员角色
- 移除成员
- 删除或转移工作区
- 管理工作区下所有项目

#### admin

权限：

- 邀请成员
- 管理普通成员
- 创建和管理项目
- 查看工作区审计日志

限制：

- 不能移除 owner
- 不应具备工作区所有权转移能力

#### member

权限：

- 查看自己有权限的项目
- 参与项目协作

限制：

- 不能管理工作区成员
- 不能修改工作区级设置

#### viewer

权限：

- 只读访问工作区中被授权的内容

限制：

- 不能编辑
- 不能评论，除非项目级单独提升

## 5. 项目权限模型

### 5.1 项目角色

- `maintainer`
- `editor`
- `commenter`
- `viewer`

### 5.2 项目角色说明

#### maintainer

权限：

- 管理项目成员
- 修改项目设置
- 编辑文件
- 评论
- 创建修订
- 回滚
- 导出

#### editor

权限：

- 编辑文件
- 评论
- 创建修订
- 导出

限制：

- 不管理项目成员

#### commenter

权限：

- 查看项目
- 基于选区发表评论
- 回复评论

限制：

- 不能修改文件
- 不能创建修订

#### viewer

权限：

- 只读查看项目内容

限制：

- 不能编辑
- 不能评论

## 6. 权限判定原则

## 6.1 基本原则

- 所有权限判断必须在服务端完成
- 前端按钮状态只能做体验优化，不能作为安全边界
- 用户必须先具备工作区访问权，才能判断项目权限
- 项目权限可以比工作区权限更细

## 6.2 判定顺序

建议统一按这个顺序校验：

1. 用户是否已登录
2. 用户是否属于当前工作区
3. 用户是否有访问当前项目的权限
4. 用户是否对当前动作具备足够角色

## 6.3 默认拒绝

对于以下情况，默认拒绝：

- 未登录
- 不属于工作区
- 不属于项目成员
- 角色不足
- 项目已归档/删除且动作不允许

## 7. 接口权限映射建议

## 7.1 用户与工作区

### 创建工作区

要求：

- 已登录

### 查看工作区成员

要求：

- 工作区成员

### 邀请成员

要求：

- `owner/admin`

### 修改成员角色

要求：

- `owner/admin`

### 移除成员

要求：

- `owner/admin`

## 7.2 项目管理

### 创建项目

要求：

- 工作区 `owner/admin/member`

### 查看项目列表

要求：

- 工作区成员

### 查看项目详情

要求：

- 项目成员

### 管理项目成员

要求：

- 项目 `maintainer`

## 7.3 文件编辑

### 查看文件树

要求：

- 项目成员

### 查看文件内容

要求：

- 项目成员

### 创建/上传/重命名/删除文件

要求：

- 项目 `maintainer/editor`

### 修改文本内容

要求：

- 项目 `maintainer/editor`

## 7.4 评论与审阅

### 发起选区评论

要求：

- 项目 `maintainer/editor/commenter`

### 回复评论

要求：

- 项目 `maintainer/editor/commenter`

### 关闭评论线程

要求：

- 评论创建者、项目 `maintainer`，或未来定义的审阅角色

## 7.5 修订与导出

### 创建修订

要求：

- 项目 `maintainer/editor`

### 回滚修订

要求：

- 项目 `maintainer`

### 导出 PDF

要求：

- 项目 `maintainer/editor`

## 8. 中间件与服务设计建议

## 8.1 推荐拆分

建议在后端实现以下辅助函数或依赖：

- `get_current_user()`
- `require_workspace_member(workspace_id)`
- `require_workspace_role(workspace_id, allowed_roles)`
- `require_project_member(project_id)`
- `require_project_role(project_id, allowed_roles)`

这样能避免权限逻辑散落在每个接口内部。

## 8.2 查询策略

建议所有受保护资源都通过“先查归属，再查权限”的方式进入：

- 项目接口先拿到 `project.workspace_id`
- 校验当前用户是否属于该工作区
- 再校验项目成员角色

## 9. 第一阶段实现步骤

建议按以下顺序落地：

1. 增加 `users` 表和密码哈希
2. 增加登录与当前用户接口
3. 增加 `workspaces`、`workspace_members`、`workspace_invites`
4. 给 `projects` 增加 `workspace_id`
5. 增加 `project_members`
6. 给现有项目接口加工作区和项目权限校验
7. 最后再扩展评论、修订、审计接口

## 10. 初期兼容策略

由于当前系统没有用户和工作区概念，第一阶段可以采用兼容式迁移：

- 允许旧项目暂时没有 `workspace_id`
- 后续通过迁移脚本把旧项目挂到默认工作区
- 在引入正式权限前，先保持现有编辑能力可运行

## 11. 安全要求

### 密码

- 只能存哈希，不能存明文
- 推荐使用成熟密码哈希方案

### 会话

- 登录态必须可失效
- 登出必须删除会话或令牌

### 接口

- 所有写操作接口都必须做鉴权和权限判断
- 不允许通过项目 ID、文件路径等手段绕过权限

## 12. 后续演进方向

第二阶段以后可以继续扩展：

- SSO
- OAuth 登录
- API Token
- 组织级审计策略
- 更细粒度权限矩阵

第一阶段不需要把这些一次做完。重点是先把“用户 -> 工作区 -> 项目 -> 权限”这条链打通。
