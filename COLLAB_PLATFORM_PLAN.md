# Typst Editor 协作平台规划

## 1. 产品目标

把当前项目从“单项目 Typst 编辑器”演进为“面向团队的在线写作平台”，核心能力包括：

- 多人协作
- 工作区与团队管理
- 项目级权限控制
- 审阅、评论与版本历史
- 稳定的预览与导出
- 易于团队自部署

目标不是“把编辑器界面继续打磨得更花”，而是“做成团队真正能用起来的文档协作系统”。

## 2. 产品定位

### 核心用户

- 论文、报告、课题文档协作的研究团队
- 需要统一模板产出的编辑或出版团队
- 维护内部 PDF 文档、手册、规范的工程团队
- 编写讲义、课程资料的教学团队

### 核心场景

- 一个团队管理多个文档项目
- 多个成员以不同角色参与同一个项目
- 审阅者对文档进行评论、跟进和关闭问题
- 团队复用模板、参考文献、图片和字体资源
- 文档完成后可稳定导出、发布或交付

## 3. 产品原则

- 团队优先：项目属于工作区，不属于匿名本地目录
- 协作优先：评论、历史、成员在线状态与编辑能力同等重要
- 输出可靠：预览与导出结果必须可预期、可复现
- 权限明确：默认一切访问都应经过权限控制
- 易于自部署：小团队应能低成本自行部署和维护

## 4. MVP 定义

第一版真正有价值的协作 MVP 建议包含：

- 用户账号体系
- 工作区创建
- 工作区成员邀请
- 项目成员与角色
- 在线文件编辑
- 项目文件树
- 项目评论
- 版本历史与回滚
- PDF 导出
- Docker Compose 部署

第一版明确不做：

- 如果会显著拖慢上线节奏，就先不做字符级实时多人协同
- 计费系统
- SSO 单点登录
- 高级模板市场
- 完整公开发布平台

## 5. 当前状态与目标状态

### 当前状态

- 以项目为中心的编辑器
- 本地或共享目录存储
- 已有文件树、预览、诊断和导出能力
- 没有用户系统
- 没有工作区和团队模型
- 没有权限系统
- 没有评论
- 没有版本历史
- 没有实时协作

### 目标状态

- 支持认证用户
- 项目归属于工作区
- 有成员、角色与权限管理
- 项目可审计、可追溯
- 支持评论与审阅流程
- 有安全可靠的协作机制
- 可部署为团队服务

## 6. 推荐架构方向

### 前端

保留现在的 React 应用，但演进为：

- 具备工作区视角的导航结构
- 项目首页或项目仪表盘
- 评论、活动流等协作侧边栏
- 在线成员与占用状态提示
- 工作区、成员、权限等设置页面

### 后端

把 API 从“项目文件操作接口”演进为一组领域服务：

- 认证服务
- 工作区服务
- 项目与成员服务
- 版本服务
- 评论服务
- 协作/在线状态服务
- 导出与部署服务

### 存储

所有元数据都应进入 PostgreSQL。文件系统只能作为实现细节，不能继续作为权威数据源。

推荐拆分方式：

- PostgreSQL：用户、工作区、成员关系、项目元数据、评论、修订记录、审计日志
- 对象存储或受控文件存储：上传资产、导出 PDF、必要时的快照文件
- 工作目录或临时卷：预览和编译过程中使用的工作副本

### 实时层

预留基于 WebSocket 的协作通道。

短期先做：

- 在线状态
- 文档占用或“软冲突”提示
- 活动更新

长期再做：

- 基于 CRDT 的实时协同编辑，优先考虑 Yjs 或同类方案

## 7. 核心领域模型

建议第一版先有这些实体：

### 身份与组织

- `users`
- `workspaces`
- `workspace_members`
- `invites`

### 项目与文件

- `projects`
- `project_members`
- `project_entries`
- `project_assets`

### 协作与审阅

- `comments`
- `comment_threads`
- `mentions`
- `presence_sessions`
- `document_locks`

### 历史与审计

- `revisions`
- `revision_entries`
- `activity_logs`

### 交付与发布

- `export_jobs`
- `deployment_targets`
- `deployment_runs`

## 8. 权限模型

一开始不要做复杂策略系统，先用角色型权限控制。

### 工作区角色

- `owner`
- `admin`
- `member`
- `viewer`

### 项目角色

- `maintainer`
- `editor`
- `commenter`
- `viewer`

### 权限示例

- 只有工作区 `owner/admin` 可以管理工作区成员
- 只有 `maintainer/editor` 可以修改项目文件
- `commenter` 可以评论，但不能修改文件
- `viewer` 只能查看，是否允许导出再单独控制

前两个阶段不需要自定义策略引擎，角色已经足够。

## 9. 协作路线

不要一开始就上最难的“完整实时协同编辑”，否则很容易拖垮整体进度。

建议按阶段推进：

### 阶段 A：非实时协作

- 项目级权限控制
- 评论与讨论线程
- 自动保存
- 版本历史
- 活动流
- “有人正在编辑此文件”的提示

### 阶段 B：增强协作

- 在线成员状态
- 当前正在查看/编辑的文件提示
- 只读光标或选区提示
- 降低误覆盖风险

### 阶段 C：实时协作

- CRDT 共享文档状态
- 光标同步
- 无冲突并发编辑
- 面向协作的版本检查点

这种分阶段方式更稳，也更容易尽快形成一个可交付产品。

## 10. 审阅流程

对在线写作平台来说，审阅能力几乎和编辑能力同等重要。

第一版审阅能力建议包括：

- 针对文件或选区发表评论
- 评论线程
- 解决/重新打开评论
- @ 提及成员
- 通过评论和项目状态实现“需要修改”这类轻量流程

后续可以增加：

- 审阅请求
- 审批流
- 修订对比
- 附带审阅摘要的导出

## 11. 版本策略

所有协作写作产品都必须有可信的历史记录。

建议最基础的版本能力包括：

- 自动保存草稿
- 显式保存检查点
- 不可变修订版本
- 回滚到指定修订
- 修订差异元数据
- 对删除、重命名等危险操作保留审计记录

推荐实现思路：

- 当前文件状态保存在 `project_entries`
- 创建检查点时写入一条 `revision`
- 把有变化的文件快照写入 `revision_entries`
- 活跃编辑期间定时生成自动检查点

## 12. 部署策略

你提到“方便团队部署”，这必须当作产品能力来设计，而不是后期补文档。

### 第一种正式支持的部署方式

单机 Docker Compose，包含：

- `web`
- `api`
- `preview`
- `compiler`
- `postgres`
- `object storage` 或本地持久化存储
- 可选的 `nginx`

### 部署要求

- 基于 `.env` 的配置方式
- 数据库持久化卷
- 文件存储持久化卷
- 备份与恢复文档
- 健康检查
- 管理员初始账号引导

### 后续再支持的部署形态

- Kubernetes
- 托管 PostgreSQL 与 S3 兼容对象存储
- 外部认证提供方

## 13. 技术里程碑

### 里程碑 1：多用户基础能力

目标：把产品从单人编辑器升级为有账号和工作区概念的平台。

交付项：

- 用户注册/登录
- 工作区模型
- 工作区成员邀请
- 项目归属到工作区
- API 基础角色校验
- 从匿名项目模型迁移出去

### 里程碑 2：团队项目管理

目标：让项目真正可共享、可治理。

交付项：

- 项目成员与角色
- 工作区范围内的归档/回收站语义
- 活动流
- 对重命名、删除、导出、成员变更等操作做审计
- 工作区与项目设置页面

### 里程碑 3：审阅与历史

目标：让协作具备基本安全性。

交付项：

- 评论与讨论线程
- 提及成员
- 自动保存与检查点
- 版本历史
- 回滚
- 文件编辑占用提示

### 里程碑 4：实时协作

目标：支持真正的多人同时编辑。

交付项：

- WebSocket 会话层
- 在线状态
- 共享文档状态
- 光标同步
- CRDT 集成

### 里程碑 5：团队部署与发布

目标：让平台具备运维和交付价值。

交付项：

- 生产环境 Compose 方案
- 备份与恢复
- 对象存储接入
- 如有需要，导出任务队列化
- 只读分享链接或发布端点

## 14. 近期工程优先级

如果现在开始推进，优先做这些：

1. 引入用户、工作区和认证体系
2. 让每个项目都归属于工作区
3. 给所有项目 API 加上成员与角色校验
4. 在动手做评论 UI 前，先设计好修订和评论表结构
5. 明确文件内容到底存数据库、文件系统还是混合存储
6. 给项目变更接口补审计日志
7. 把前端路由从“项目列表 -> 编辑器”升级为“工作区 -> 项目 -> 编辑器”

## 15. 建议的 4 周执行计划

### 第 1 周

- 明确 PRD 和范围
- 完成数据库模型设计
- 接入基础认证
- 增加工作区与成员模型

### 第 2 周

- 把项目 API 改成工作区归属
- 增加权限检查
- 增加工作区/项目设置 UI
- 实现邀请流程

### 第 3 周

- 实现版本检查点
- 实现评论与线程
- 增加活动流 UI/API

### 第 4 周

- 增加在线状态和正在编辑提示
- 整理生产部署方案
- 编写部署与备份文档
- 稳定化与测试

## 16. 主要风险

### 风险：过早上 CRDT

影响：

- 会拖慢其他核心能力
- 会引入大量同步一致性问题

应对：

- 先上线审阅、历史、在线状态和占用提示

### 风险：继续以文件系统为中心

影响：

- 难以扩展
- 审计和版本能力很难做好

应对：

- 现在就把权威元数据和修订记录转入 PostgreSQL

### 风险：权限边界不清晰

影响：

- 团队产品会很快变得不安全

应对：

- 在继续加协作功能之前，先把 API 级权限控制做完整

### 风险：部署方式过于随意

影响：

- 团队无法放心采用

应对：

- 尽早确定并维护一条官方支持的部署路径

## 17. MVP 成功标准

如果一个小团队能够做到以下事情，就说明 MVP 已经成立：

- 创建工作区
- 邀请成员
- 创建并共享项目
- 在线编辑项目文件
- 留言评论并解决评论
- 查看版本历史
- 导出 PDF
- 按文档说明完成自部署
